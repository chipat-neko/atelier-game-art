U10 — Plugins & packages

Débutant 30 min Piste 3 · Unity
PrérequisU2 — Éditeur.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Ouvrir et lire le Package Manager (Window▸Package Manager) : onglets In Project, Unity Registry, My Assets, Built-in.
  • Installer, mettre à jour et désinstaller un package proprement, et lire ses dépendances.
  • Comprendre le fichier Packages/manifest.json et le versionnage sémantique (SemVer).
  • Distinguer les sources : Unity Registry, Asset Store (My Assets), Git URL, disque local, et scoped registries (OpenUPM).
  • Acheter/importer un asset depuis l'Asset Store et comprendre ce qu'est un .unitypackage.
  • Reconnaître les packages utiles au graphiste : ProBuilder, Cinemachine, FBX Exporter, Shader Graph, VFX Graph.
  • Anticiper les conflits de versions, éviter de polluer un projet, et reconnaître un plugin abandonné ou incompatible.

Pourquoi c'est important

Unity « nu » ne sait pas tout faire. Le moteur livré par défaut est volontairement minimal : il contient le rendu, la physique, l'éditeur de base, mais pas les caméras cinématiques, pas l'outil de modélisation in-editor, pas le système d'effets visuels nodal. Tout cela vit dans des packages — des modules que tu ajoutes à la demande, projet par projet. Savoir gérer ces packages, c'est savoir composer ton Unity comme une boîte à outils : tu prends exactement ce dont la production a besoin, ni plus, ni moins.

Côté graphisme, c'est loin d'être accessoire. Le système de pipeline de rendu URP ou HDRP est un package ; Shader Graph (créer des shaders sans code) est un package ; VFX Graph (particules sur GPU) est un package ; ProBuilder (bloquer un niveau directement dans l'éditeur) est un package ; Cinemachine (caméras intelligentes) est un package. Si tu ne maîtrises pas le Package Manager, tu ne peux littéralement pas activer la moitié des outils graphiques de Unity.

Mais l'enjeu est aussi celui de la propreté du projet. En production, un projet qui s'alourdit de plugins mal choisis devient lent à ouvrir, lourd à versionner, et fragile : un seul plugin abandonné peut bloquer une mise à jour de Unity et faire perdre des jours à toute l'équipe. Apprendre à ajouter proprement, c'est aussi apprendre à ne pas ajouter n'importe quoi — la discipline qui sépare un projet maintenable d'un dépotoir.

Théorie en profondeur

1. Qu'est-ce qu'un package, au juste ?

Un package est un dossier autonome de fichiers (scripts, shaders, assets, documentation) accompagné d'un manifeste qui décrit son nom, sa version et ses dépendances. Quand tu « installes » un package, Unity ne copie pas forcément des fichiers dans ton dossier Assets/ : la plupart du temps il les place dans un cache partagé sur ta machine et y fait simplement référence. C'est une différence majeure avec l'ancien monde des .unitypackage, où tout était déversé en vrac dans le projet.

Le système qui orchestre cela s'appelle UPM (Unity Package Manager, « gestionnaire de paquets Unity »). L'analogie la plus juste vient du développement logiciel : UPM est à Unity ce que npm est à JavaScript ou pip est à Python. Tu déclares « je veux la bibliothèque X en version Y », et le gestionnaire se charge de la télécharger, de résoudre ses dépendances, et de la rendre disponible. Tu ne gères plus des fichiers : tu gères des déclarations de besoin.

ℹ️Package géré (UPM) vs asset importé

Retiens deux mondes. Un package UPM (URP, Cinemachine, un dépôt Git) est référencé et reste à l'extérieur de Assets/ : tu peux le désinstaller en un clic sans laisser de trace. Un asset importé depuis un .unitypackage (souvent l'Asset Store) copie ses fichiers dans Assets/ : il devient « dans » ton projet, et le retirer proprement est bien plus pénible. Cette distinction explique pourquoi on privilégie les packages UPM quand on a le choix.

2. Le Package Manager : anatomie de la fenêtre

On ouvre la fenêtre par Window▸Package Manager. En haut à gauche, un menu déroulant Packages: filtre la source affichée. C'est le réglage le plus important de la fenêtre ; il change complètement ce que tu vois.

  • In Project — ce qui est réellement installé dans ce projet. C'est ta vérité : si un outil n'apparaît pas ici, il n'est pas disponible dans la scène. La liste se divise souvent en Packages - Unity et Custom.
  • Unity Registry — le catalogue officiel de Unity : tous les packages maintenus par Unity (URP, HDRP, Shader Graph, VFX Graph, Cinemachine, ProBuilder, Input System, Timeline, FBX Exporter…). C'est ici que tu vas chercher 90 % de ce dont un graphiste a besoin.
  • My Assets — ta bibliothèque Asset Store : tout ce que tu as acheté ou récupéré gratuitement avec ton compte Unity, prêt à être téléchargé puis importé.
  • Built-in — les modules internes du moteur (physique, audio, terrain, particules « legacy »…). On les active/désactive rarement, mais les couper allège un build minimaliste.

À droite, le panneau de détail affiche pour le package sélectionné : sa description, sa version et le sélecteur de versions, ses dépendances (les autres packages qu'il tire derrière lui), un onglet Samples (exemples optionnels à importer), et les boutons d'action Install / Update / Remove. Le verbe change selon l'état : Install si absent, Update to x.y.z si une version plus récente existe, Remove si présent.

3. Les sources : d'où viennent les packages

Comprendre les sources, c'est comprendre la confiance et le risque que chacune implique.

Unity Registry — le terrain sûr

C'est le catalogue maintenu par Unity. Les packages y sont testés contre la version du moteur, versionnés proprement, et leurs dépendances sont garanties cohérentes. Quand le Registry te propose une version « verified » ou recommandée pour ta version d'éditeur, c'est celle qui a été validée : prends-la par défaut. C'est la source que tu utiliseras le plus.

Asset Store / My Assets — le marché

L'Asset Store (assetstore.unity.com, dans le navigateur) est la place de marché de Unity : modèles 3D, textures, shaders, outils, musiques. On y « achète » (gratuit ou payant) avec son compte Unity ; l'asset rejoint alors la bibliothèque du compte. De retour dans l'éditeur, l'onglet My Assets du Package Manager liste cette bibliothèque : on y télécharge puis on importe. Beaucoup d'assets du Store sont encore distribués en .unitypackage — ils copient leurs fichiers dans Assets/, d'où l'importance d'importer sélectivement (voir plus bas).

Git URL — l'outil de la communauté

Énormément d'outils gratuits de la communauté vivent sur GitHub. UPM sait installer un package directement depuis l'URL d'un dépôt Git : pas besoin de télécharger un zip à la main, ni de copier des fichiers. Tu colles l'adresse du dépôt, Unity le clone et l'ajoute comme package référencé. C'est puissant et propre — à condition d'épingler une version (voir SemVer) pour ne pas te réveiller un matin avec un commit qui casse tout.

Disque local — le développement

On peut pointer un package vers un dossier local de la machine. C'est le mode du développement d'outils maison : tu travailles sur le package dans un dossier, et le projet le voit en direct. Utile pour partager un package interne entre plusieurs projets sans le republier à chaque modification.

Scoped registries (OpenUPM) — l'écosystème ouvert

Une scoped registry est un registre tiers que tu déclares en plus du Registry officiel, avec une « portée » (les préfixes de noms qu'il fournit). Le plus connu est OpenUPM, qui héberge des centaines de packages open source à la manière de npm. Une fois la registry ajoutée dans les Project Settings ▸ Package Manager, ses packages apparaissent comme s'ils étaient natifs. C'est confortable, mais cela ajoute une dépendance à un service externe : à garder à l'esprit pour la pérennité.

4. Le manifest.json et le versionnage sémantique

Tout ce que tu fais dans le Package Manager se traduit, en coulisses, par l'écriture d'un fichier texte : Packages/manifest.json. C'est la source de vérité du projet : la liste de tout ce qu'il réclame, avec les versions exactes. Comprendre ce fichier, c'est cesser d'avoir peur du Package Manager — il n'y a aucune magie, juste une liste.

{
  "dependencies": {
    "com.unity.render-pipelines.universal": "14.0.9",
    "com.unity.cinemachine": "2.9.7",
    "com.unity.probuilder": "5.1.1",
    "com.unity.shadergraph": "14.0.9",
    "com.mon-outil.super-tool": "https://github.com/auteur/super-tool.git#1.2.0"
  },
  "scopedRegistries": [
    {
      "name": "OpenUPM",
      "url": "https://package.openupm.com",
      "scopes": [ "com.cysharp" ]
    }
  ]
}

Chaque ligne de dependencies associe un nom inversé (style com.unity.cinemachine, garanti unique) à une version. La version suit le versionnage sémantique (SemVer), un standard universel en trois nombres : MAJEUR.MINEUR.CORRECTIF (par ex. 14.0.9).

SegmentExempleCe que ça signifie — et pourquoi t'en soucier
MAJEUR14.x.xChangement cassant : l'API change, du code qui marchait peut ne plus compiler. Passer de 13 à 14 demande des tests.
MINEURx.2.xNouvelles fonctions compatibles : rien ne casse, on gagne des features. Mise à jour généralement sûre.
CORRECTIFx.x.9Corrections de bugs uniquement. Très sûr : on prend presque toujours le dernier correctif.
💡Le manifest se modifie au Package Manager, pas à la main

On peut éditer manifest.json dans un éditeur de texte (utile pour ajouter une scoped registry vite fait), mais 99 % du temps tu passes par la fenêtre : elle valide la syntaxe et résout les dépendances pour toi. Une virgule en trop ou une accolade manquante dans ce fichier, et Unity refuse d'ouvrir le projet : garde-le sous contrôle de version pour pouvoir revenir en arrière.

5. Dépendances et conflits : le cœur du sujet

Un package en tire souvent d'autres. Quand tu installes URP, tu n'installes pas qu'URP : il dépend de Shader Graph et de Core RP Library, qui s'installent automatiquement. C'est le rôle central d'UPM : il lit l'arbre de dépendances et tire tout ce qu'il faut, sans que tu aies à le savoir.

Le problème surgit quand deux packages réclament le même package en versions différentes. Imagine que ton outil A exige Core RP Library 13, mais que ton URP fraîchement installé exige Core RP Library 14. UPM doit arbitrer : il choisit en général la version la plus élevée compatible, mais si A ne supporte vraiment pas la 14, tu obtiens des erreurs de compilation. C'est le fameux « conflit de versions » — la première cause de projets qui ne compilent plus après l'ajout d'un plugin.

6. Garder à jour… mais pas n'importe comment

Mettre à jour n'est pas une fin en soi. La bonne discipline :

  • Correctifs (dernier chiffre) — prends-les, ils corrigent des bugs sans risque.
  • Mineures (chiffre du milieu) — généralement sûres ; teste vite après coup.
  • Majeures (premier chiffre) — jamais en plein milieu d'une production sans une fenêtre de test. Une majeure peut renommer des API, déplacer des réglages, casser des shaders. On la planifie.

Et la règle d'or, valable côté graphisme comme code : aligner les versions des packages d'un même groupe. URP, Shader Graph, VFX Graph et Core RP Library appartiennent à la même famille SRP ; ils doivent tous être dans la même version majeure que ta version d'éditeur. Mélanger URP 14 et Shader Graph 12, c'est l'assurance de shaders qui ne se compilent pas.

🔧 Démonstration pas à pas

Fil rouge : installer ProBuilder depuis le Registry, puis ajouter un outil via une Git URL, et vérifier l'effet sur le manifest.json.

  1. Ouvrir le gestionnaire. Window▸Package Manager. La fenêtre s'ouvre ; en haut à gauche, le menu Packages: est sans doute sur In Project — tu vois ce qui est déjà installé.
  2. Passer au catalogue Unity. Clique sur Packages: et choisis Unity Registry. La liste se remplit du catalogue officiel. Utilise la barre de recherche en haut et tape ProBuilder.
  3. Lire avant d'installer. Sélectionne ProBuilder : le panneau de droite affiche sa description, la version proposée, et la section Dependencies. C'est l'étape que les débutants sautent : regarde ce qui va être tiré derrière.
  4. Installer. Clique sur Install (en bas à droite). Unity télécharge ProBuilder et ses dépendances, met à jour manifest.json, puis recompile. ProBuilder bascule alors dans In Project et un menu Tools▸ProBuilder apparaît dans la barre de menus : c'est la preuve que l'outil est actif.
  5. Importer un exemple (optionnel). Toujours sur la fiche ProBuilder, déroule Samples s'il y en a et clique Import : les exemples atterrissent dans Assets/Samples/. Ils sont optionnels et tu peux les supprimer ensuite ; ne les importe que si tu veux étudier un cas.
  6. Ajouter un package par Git URL. En haut à gauche, clique sur le bouton + puis Add package from git URL…. Un champ apparaît : colle l'adresse .git du dépôt, idéalement en épinglant une version avec #1.2.0 à la fin (le tag ou la branche). Valide avec Add.
  7. Observer la résolution. Unity clone le dépôt, lit son manifeste, tire ses dépendances et recompile. Le nouvel outil apparaît dans In Project ▸ Custom. Si une erreur s'affiche dans la Console, c'est souvent une incompatibilité de version : lis le message, il nomme le package fautif.
  8. Vérifier le manifest. Ouvre Packages/manifest.json (dans l'Explorateur, à côté de Assets/). Tu y vois deux nouvelles lignes : com.unity.probuilder avec un numéro de version, et ton outil Git avec l'URL complète. Tu viens de visualiser exactement ce que tes clics ont écrit.
  9. Désinstaller proprement. Pour retirer un package : sélectionne-le dans In Project et clique Remove. La ligne disparaît du manifest, le cache n'est pas touché, et le projet redevient propre — aucun fichier orphelin dans Assets/. C'est tout l'intérêt d'un package UPM par rapport à un .unitypackage.

⚠️ Pièges & erreurs courantes

SymptômeMon projet a triplé de taille et met une éternité à s'ouvrir après un import.
CauseJ'ai importé un asset Asset Store géant en entier (démos, scènes, textures 4K), qui a déversé des milliers de fichiers dans Assets/.
CorrectionÀ l'import du .unitypackage, décoche les dossiers de démos/exemples et ne garde que ce que tu utilises. Supprime ensuite les dossiers d'exemples inutiles.
SymptômeAprès avoir ajouté un plugin, le projet ne compile plus : « version conflict ».
CauseUn conflit de dépendances : deux packages exigent la même bibliothèque (souvent Core RP) en versions incompatibles.
CorrectionLis le message de la Console ; il nomme le package et les versions. Aligne les versions, ou retire le plugin fautif. Sauvegarde le manifest avant d'expérimenter.
SymptômeUn outil Git installé « marchait » et un jour il a cassé tout seul.
CauseLa Git URL pointait sur une branche (pas un tag) : un nouveau commit en amont a changé l'outil sans prévenir.
CorrectionÉpingle une version précise avec #1.2.0 à la fin de l'URL. Tu figes l'outil ; tu décides quand monter de version.
SymptômeUnity refuse d'ouvrir le projet et parle d'erreur dans manifest.json.
CauseLe manifest a été cassé : virgule en trop, accolade manquante après une édition à la main.
CorrectionRestaure la dernière version saine via ton contrôle de version (Git). Édite ce fichier avec un éditeur qui valide le JSON, ou passe par le Package Manager.
SymptômeLe plugin star de mon projet est incompatible avec la nouvelle version d'Unity.
CausePlugin abandonné par son auteur : plus de mises à jour, il ne suit pas les évolutions du moteur.
CorrectionAvant d'adopter un plugin, vérifie sa date de dernière mise à jour et les versions d'Unity supportées. Préfère un équivalent maintenu ; évite d'en faire un pilier critique.
SymptômeUn coéquipier ne peut pas ouvrir le projet : il manque un asset payant.
CauseDépendance à un asset payant non versionné lié à ton compte : il n'est ni dans le dépôt, ni dans la bibliothèque du coéquipier.
CorrectionDocumente les assets payants requis et les licences. Vérifie que chaque membre y a accès, ou substitue une solution libre pour les dépendances critiques.

🔧Exercice pratique

Sur un projet de test (jamais sur une vraie production), prends en main le Package Manager de bout en bout :

  1. Note la taille du dossier projet et le contenu de Packages/manifest.json avant toute action (capture ou copie).
  2. Installe ProBuilder depuis le Unity Registry en lisant d'abord ses dépendances dans le panneau de droite.
  3. Crée un cube ProBuilder dans la scène (Tools▸ProBuilder▸New Shape) pour vérifier que l'outil est bien actif.
  4. Ajoute un package via Git URL en épinglant une version avec # à la fin de l'adresse.
  5. Ouvre manifest.json et compare avec l'état initial : identifie les lignes ajoutées et explique chacune.
  6. Désinstalle ProBuilder via Remove et vérifie que Assets/ ne contient aucun fichier orphelin.
Critères de réussite
  • Tu peux pointer, dans le manifest, la ligne ajoutée par ProBuilder et celle ajoutée par la Git URL.
  • Ta Git URL contient un #version et tu sais expliquer pourquoi ce n'est pas une branche.
  • Tu sais nommer au moins une dépendance tirée automatiquement par un package installé.
  • Après Remove, le projet compile sans erreur et aucun dossier orphelin ne traîne dans Assets/.
  • Tu sais expliquer la différence entre un package UPM et un .unitypackage de l'Asset Store.

Glossaire des termes introduits

Package — module autonome de fichiers (scripts, shaders, assets) avec un manifeste décrivant son nom, sa version et ses dépendances, ajouté à la demande dans un projet Unity.

UPM (Unity Package Manager) — système qui télécharge, référence et résout les packages d'un projet, à la manière de npm ou pip. La fenêtre porte le même nom.

Unity Registry — catalogue officiel des packages maintenus et testés par Unity (URP, Cinemachine, ProBuilder, Shader Graph…).

Asset Store — place de marché de Unity (assetstore.unity.com) où l'on acquiert des assets, retrouvés ensuite dans l'onglet My Assets du Package Manager.

.unitypackage — archive d'assets qui copie ses fichiers directement dans Assets/ à l'import, par opposition à un package UPM seulement référencé.

Git URL — adresse d'un dépôt Git que le Package Manager peut installer directement comme package, idéalement épinglée à une version avec #x.y.z.

Scoped registry — registre tiers déclaré en plus du Registry officiel (ex. OpenUPM) qui fournit des packages selon une portée de noms.

manifest.json — fichier texte (Packages/manifest.json) listant toutes les dépendances et versions du projet : sa source de vérité.

Versionnage sémantique (SemVer) — convention de version en trois nombres MAJEUR.MINEUR.CORRECTIF indiquant la nature des changements (cassant, compatible, correctif).

Dépendance — package qu'un autre réclame pour fonctionner et qu'UPM installe automatiquement (ex. URP qui tire Shader Graph et Core RP Library).

Tous ces termes (et bien d'autres) sont rassemblés dans le glossaire global.
