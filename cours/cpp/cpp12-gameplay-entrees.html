CPP12 — Gameplay framework & entrées

Avancé 45 min Piste 8 · C++ (Unreal)
PrérequisCPP9 — Actors et CPP7.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Identifier les rôles de GameMode, Pawn/Character, PlayerController.
  • Comprendre la possession d'un Pawn par un Controller.
  • Mettre en place l'Enhanced Input (Mapping Context, Input Action).
  • Brancher une entrée sur une fonction C++ pour déplacer un personnage.

Pourquoi c'est important

Unreal sépare proprement « qui joue » (le Controller), « quel corps » (le Pawn/Character) et « quelles règles » (le GameMode). Cette architecture, déroutante au début, rend le code réutilisable et réseau-compatible. Et depuis UE5, les entrées passent par l'Enhanced Input, bien plus souple que l'ancien système.

Théorie en profondeur

1. Les pièces du gameplay framework

ClasseRôle
AGameModeBaseLes règles : quel Pawn par défaut, conditions de victoire, qui peut rejoindre.
APawn / ACharacterLe corps contrôlable. ACharacter = Pawn avec déplacement bipède intégré.
APlayerControllerLe cerveau du joueur : reçoit les entrées, possède un Pawn.
APlayerStateDonnées persistantes du joueur (score, nom).

La possession : un Controller « possède » un Pawn pour le piloter. On peut changer de Pawn (entrer dans un véhicule) sans changer de Controller.

2. Un Character en C++

// Heros.h
UCLASS()
class MONJEU_API AHeros : public ACharacter
{
    GENERATED_BODY()
public:
    AHeros();
protected:
    virtual void SetupPlayerInputComponent(UInputComponent* PlayerInputComponent) override;
    void Deplacer(const FInputActionValue& Value);   // recevra l'entrée

    UPROPERTY(EditAnywhere, Category = "Input")
    UInputMappingContext* ContexteDefaut;

    UPROPERTY(EditAnywhere, Category = "Input")
    UInputAction* ActionDeplacer;
};

3. Enhanced Input : Mapping Context & Input Action

Deux assets à créer dans l'éditeur :

  • Input Action (IA) : une intention abstraite (« Déplacer », « Sauter »), avec un type de valeur (Axis2D, booléen…).
  • Input Mapping Context (IMC) : relie des touches physiques aux Input Actions (ZQSD → Déplacer).

On active le contexte pour le joueur, puis on lie chaque action à une fonction :

void AHeros::SetupPlayerInputComponent(UInputComponent* PlayerInputComponent)
{
    Super::SetupPlayerInputComponent(PlayerInputComponent);

    // Active le contexte d'entrées pour ce joueur
    if (APlayerController* PC = Cast<APlayerController>(GetController()))
    {
        if (UEnhancedInputLocalPlayerSubsystem* Sub =
            ULocalPlayer::GetSubsystem<UEnhancedInputLocalPlayerSubsystem>(PC->GetLocalPlayer()))
        {
            Sub->AddMappingContext(ContexteDefaut, 0);
        }
    }

    // Lie l'action "Déplacer" à notre fonction
    if (UEnhancedInputComponent* EIC = Cast<UEnhancedInputComponent>(PlayerInputComponent))
    {
        EIC->BindAction(ActionDeplacer, ETriggerEvent::Triggered, this, &AHeros::Deplacer);
    }
}
void AHeros::Deplacer(const FInputActionValue& Value)
{
    const FVector2D Axe = Value.Get<FVector2D>();   // X = latéral, Y = avant
    AddMovementInput(GetActorForwardVector(), Axe.Y);
    AddMovementInput(GetActorRightVector(),   Axe.X);
}

Cast<T> convertit un pointeur vers un type plus précis (renvoie nullptr si la conversion échoue — d'où les if). AddMovementInput utilise le composant de déplacement du ACharacter.

🔧 Démonstration pas à pas

  1. Crée un ACharacter C++ AHeros et un AGameModeBase qui le définit comme Pawn par défaut.
  2. Dans l'éditeur, crée une Input Action « Déplacer » (Axis2D) et un Mapping Context (ZQSD/WASD).
  3. Assigne ces assets aux UPROPERTY du héros, ajoute le contexte et lie l'action dans SetupPlayerInputComponent.
  4. Implémente Deplacer avec AddMovementInput et teste.

⚠️ Pièges & erreurs courantes

SymptômeAucune entrée ne répond.
CauseMapping Context jamais ajouté au subsystem.
CorrectionAppelle AddMappingContext dans SetupPlayerInputComponent (ou BeginPlay).
Symptôme« EnhancedInput introuvable » à la compilation.
CauseModule non déclaré.
CorrectionAjoute "EnhancedInput" dans .Build.cs (dépendances).
SymptômeLe personnage n'est pas contrôlé.
CausePawn non possédé / GameMode non configuré.
CorrectionDéfinis le GameMode et son DefaultPawnClass, ou « Auto Possess Player ».
SymptômeCrash sur un Cast.
CauseRésultat du Cast non vérifié (nullptr).
CorrectionEncadre par if (T* p = Cast<T>(...)).

🔧Exercice pratique

  1. Un ACharacter déplaçable (Axis2D) via Enhanced Input.
  2. Ajoute une action « Sauter » (booléenne) liée à ACharacter::Jump.
  3. Configure un GameMode qui spawn ton héros par défaut.
Critères de réussite
  • Le personnage se déplace et saute via les Input Actions.
  • Le module EnhancedInput est bien déclaré.
  • Chaque Cast est vérifié.
📌À retenir
  • Rôles séparés : GameMode (règles), Pawn/Character (corps), PlayerController (entrées).
  • Un Controller possède un Pawn pour le piloter.
  • Enhanced Input : Input Action (intention) + Mapping Context (touches), ajouté au subsystem.
  • BindAction relie une action à une fonction ; AddMovementInput déplace le Character.

Glossaire des termes introduits

GameMode — définit les règles de la partie et le Pawn par défaut.

Pawn / Character — corps contrôlable ; Character ajoute le déplacement bipède.

PlayerController — reçoit les entrées et possède un Pawn.

Enhanced Input — système d'entrées UE5 (Input Action + Mapping Context).

Cast — conversion sûre d'un pointeur vers un type plus précis (nullptr si échec).
