CPP10 — Pointeurs & mémoire (GC Unreal)

Avancé 45 min Piste 8 · C++ (Unreal)
PrérequisCPP8 et CPP9.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Comprendre un pointeur, nullptr et le déréférencement.
  • Comprendre le ramasse-miettes (GC) d'Unreal.
  • Garder un objet en vie avec UPROPERTY() / TObjectPtr.
  • Utiliser TWeakObjectPtr et IsValid pour éviter les accès invalides.

Pourquoi c'est important

La mémoire est la grande différence avec le C#. En C++ pur, on gère soi-même la durée de vie des objets — source de bugs redoutés (fuites, accès à un objet détruit). Unreal apporte un ramasse-miettes pour ses UObject, mais il faut comprendre ses règles, sinon tes objets disparaissent… ou ne disparaissent jamais.

Théorie en profondeur

1. Pointeur vs référence

Un pointeur (Type*) contient l'adresse d'un objet — il peut être nul (nullptr) et changer de cible. Une référence (Type&, vue en CPP6) est un alias qui doit toujours désigner un objet valide.

AActor* Cible = nullptr;       // pointeur nul : ne pointe sur rien
Cible = AutreActor;            // pointe maintenant sur AutreActor
if (Cible != nullptr)          // TOUJOURS vérifier avant de déréférencer
{
    Cible->Destroy();          // -> accède à un membre via un pointeur
}

Déréférencer un pointeur nul = crash. La vérification avant accès est un réflexe vital en C++.

2. Le ramasse-miettes (GC) d'Unreal

Tous les UObject (dont les AActor) sont gérés par un ramasse-miettes : périodiquement, le moteur détruit ceux qui ne sont plus référencés. La règle d'or : seules les UPROPERTY() comptent comme références. Un pointeur d'UObject sans UPROPERTY() est ignoré du GC — l'objet peut être détruit sous tes pieds, laissant un pointeur « pendouillant ».

UPROPERTY()                       // ✅ le GC voit cette référence → objet maintenu en vie
TObjectPtr<AArme> ArmeEquipee;     // UE5 : forme moderne, équivalente à AArme*

AArme* ArmeBrute;                 // ❌ sans UPROPERTY : invisible au GC, dangereux

En UE5, on préfère TObjectPtr<T> à T* pour les membres : même rôle, mais avec des vérifications supplémentaires dans l'éditeur.

3. Références faibles : TWeakObjectPtr

Parfois on veut pointer sans empêcher la destruction (ex. : la cible d'une IA, qui peut mourir). Un TWeakObjectPtr n'empêche pas le GC ; on vérifie sa validité avant usage :

UPROPERTY()
TWeakObjectPtr<AActor> CibleIA;

void Attaquer()
{
    if (CibleIA.IsValid())          // la cible existe-t-elle encore ?
    {
        CibleIA->TakeDamage(/*...*/);
    }
}

4. IsValid : la vérification universelle

IsValid(Obj) renvoie vrai si l'objet n'est ni nul ni en cours de destruction. C'est plus sûr qu'un simple != nullptr pour un UObject :

if (IsValid(ArmeEquipee))
{
    ArmeEquipee->Tirer();
}
ℹ️Et les objets non-UObject ?

Pour des classes C++ ordinaires (préfixe F, hors UObject), le GC ne s'applique pas. On utilise alors les smart pointers de la STL/Unreal : TSharedPtr (possession partagée, libère quand plus personne ne pointe) et TUniquePtr (possession unique). On évite presque toujours new/delete bruts.

🔧 Démonstration pas à pas

  1. Déclare UPROPERTY() TObjectPtr<AArme> ArmeEquipee; et équipe une arme par SpawnActor.
  2. Supprime volontairement la UPROPERTY() et observe (en jeu prolongé) la disparition de l'arme : le GC l'a collectée.
  3. Stocke une cible d'IA en TWeakObjectPtr ; détruis la cible et vérifie IsValid() avant d'attaquer.
  4. Protège chaque déréférencement par if (IsValid(...)).

🔧Exercice pratique

  1. Un composant d'inventaire qui garde des armes dans des UPROPERTY() (donc maintenues en vie).
  2. Une IA dont la cible est un TWeakObjectPtr ; gère proprement la mort de la cible.
  3. Encadre tous les accès par pointeur avec IsValid.
Critères de réussite
  • Aucun objet attendu ne disparaît (références correctes).
  • Aucun crash sur pointeur nul/détruit.
  • Les cibles potentiellement détruites sont en référence faible.

📌À retenir
  • Un pointeur peut être nullptr : vérifie avant de déréférencer.
  • Le GC conserve les UObject référencés par une UPROPERTY() (ou TObjectPtr).
  • TWeakObjectPtr + IsValid() pour pointer sans empêcher la destruction.
  • Objets non-UObject : TSharedPtr/TUniquePtr, pas new/delete bruts.

Glossaire des termes introduits

Pointeur / nullptr — variable contenant une adresse / pointeur ne désignant rien.

Ramasse-miettes (GC) — système Unreal détruisant les UObject non référencés.

TObjectPtr — forme moderne (UE5) d'un pointeur d'UObject membre, avec vérifications.

TWeakObjectPtr — référence faible n'empêchant pas la destruction (testée par IsValid).

IsValid — teste qu'un objet n'est ni nul ni en destruction.
