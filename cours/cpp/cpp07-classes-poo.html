CPP07 — Classes & POO en C++

Intermédiaire 45 min Piste 8 · C++ (Unreal)
PrérequisCPP6 — Fonctions & en-têtes.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Définir une classe avec ses membres et ses niveaux d'accès.
  • Écrire un constructeur (et comprendre le destructeur).
  • Appliquer l'encapsulation (public/private/protected).
  • Utiliser l'héritage, virtual/override et le polymorphisme.

Pourquoi c'est important

La POO structure le code autour d'objets qui regroupent données et comportements. Toute l'architecture d'Unreal en découle : AActor, APawn, ACharacter forment une chaîne d'héritage. Maîtriser la POO C++ permet de comprendre et d'étendre le moteur.

Théorie en profondeur

1. Définir une classe

class FPersonnage          // 'F' : structure de données simple (hors UObject)
{
public:                    // accessible de l'extérieur
    FPersonnage(const FString& NomInit, int32 PVInit);  // constructeur
    void SubirDegats(int32 Montant);
    int32 GetPV() const;   // accesseur en lecture

private:                   // interne, protégé de l'extérieur
    FString Nom;
    int32 PV;
};

Les spécificateurs d'accès : public (visible partout), private (seulement dans la classe), protected (classe + classes filles). C'est l'encapsulation : on cache les données et on expose une interface contrôlée.

2. Constructeur & destructeur

// Constructeur : initialise l'objet à sa création (liste d'initialisation)
FPersonnage::FPersonnage(const FString& NomInit, int32 PVInit)
    : Nom(NomInit), PV(PVInit)       // initialise les membres
{
}

void FPersonnage::SubirDegats(int32 Montant)
{
    PV -= Montant;
    if (PV < 0) { PV = 0; }
}

int32 FPersonnage::GetPV() const { return PV; }

Le destructeur (~FPersonnage()) est appelé à la destruction pour libérer d'éventuelles ressources. Pour les classes Unreal (UObject/AActor), on n'écrit presque jamais de destructeur : le moteur s'en charge (voir CPP10).

3. Héritage

Une classe peut hériter d'une autre : elle reprend ses membres et peut les étendre. C'est exactement ce que fait Unreal :

// AEnnemi hérite d'AActor : il EST un Actor, plus ses propres ajouts
UCLASS()
class MONJEU_API AEnnemi : public AActor
{
    GENERATED_BODY()
public:
    AEnnemi();
    virtual void BeginPlay() override;   // on REDÉFINIT une fonction du parent
};

4. virtual, override & polymorphisme

Une fonction virtual peut être redéfinie par une classe fille avec override. Le polymorphisme : via un pointeur de type parent, c'est la version de l'objet réel qui s'exécute :

class AArme : public AActor
{
public:
    virtual void Tirer();           // comportement par défaut
};

class AFusil : public AArme
{
public:
    virtual void Tirer() override;  // version spécialisée
};

// Polymorphisme : on manipule par le type de base...
AArme* Arme = SpawnFusil();
Arme->Tirer();   // ...mais c'est AFusil::Tirer() qui s'exécute

Sans virtual, c'est la version du type du pointeur qui serait appelée — le polymorphisme ne fonctionnerait pas. Toujours mettre override : le compilateur vérifie que tu redéfinis bien une fonction existante.

🖐Essaie maintenant~1 min

Pour class Arme { public: int32 Degats = 10; void Tirer(); };, où mets-tu la déclaration et l'implémentation ? (déclaration dans le .h, corps de Tirer() dans le .cpp.)

🔧 Démonstration pas à pas

  1. Écris une classe FPersonnage avec Nom/PV privés, un constructeur et SubirDegats/GetPV.
  2. Crée une classe Unreal AEnnemi : public AActor et override BeginPlay().
  3. Ajoute une fonction virtual void Attaquer() et une classe fille qui la spécialise avec override.
  4. Manipule la classe fille via un pointeur de type parent et observe le polymorphisme.

⚠️ Pièges & erreurs courantes

SymptômeC'est la version du parent qui s'exécute.
CauseFonction non déclarée virtual dans le parent.
CorrectionMets virtual au parent et override à l'enfant.
Symptôme« cannot access private member ».
CauseAccès direct à un membre private depuis l'extérieur.
CorrectionPasse par un accesseur public, ou rends le membre protected.
SymptômeComportement de base oublié dans l'override.
CauseTu n'appelles pas Super::Fonction().
CorrectionAppelle Super::BeginPlay(); en début d'override Unreal.
SymptômeErreur « override ne correspond à rien ».
CauseSignature différente de la fonction parente.
CorrectionRecopie exactement la signature (types, const).

🔧Exercice pratique

  1. Une classe de base AArme avec virtual void Tirer() et une fille AFusilAPompe qui l'override.
  2. Encapsule les munitions en privé ; expose Recharger() et GetMunitions().
  3. Stocke plusieurs armes dans un TArray<AArme*> et appelle Tirer() sur chacune (polymorphisme).
Critères de réussite
  • Les données sensibles sont private avec des accesseurs.
  • L'override fonctionne et appelle Super:: si nécessaire.
  • Le polymorphisme exécute la bonne version par arme.

📌À retenir
  • Une classe regroupe données + comportements ; public/private/protected = encapsulation.
  • Le constructeur initialise (liste d'initialisation) ; en Unreal, le moteur gère la destruction.
  • Héritage = relation « est-un(e) » ; : public Base.
  • virtual + override = polymorphisme ; appelle Super:: pour garder le comportement parent.

Glossaire des termes introduits

Encapsulation — cacher les données (private) et exposer une interface contrôlée.

Constructeur / destructeur — fonctions appelées à la création / destruction d'un objet.

virtual / override — autorise et signale la redéfinition d'une fonction par une classe fille.

Polymorphisme — via un pointeur de base, exécution de la version de l'objet réel.

Super — alias Unreal de la classe parente (pour appeler sa version d'une fonction).
