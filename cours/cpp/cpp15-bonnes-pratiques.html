CPP15 — Bonnes pratiques, performances & organisation

Avancé 50 min Piste 8 · C++ (Unreal)
Prérequisidéalement toute la piste, surtout CPP9, CPP10 et CPP13.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Appliquer les conventions Unreal (préfixes, nommage).
  • Maîtriser le Tick et préférer événements/timers.
  • Accélérer la compilation (forward declarations, includes).
  • Déboguer avec UE_LOG et réaliser un mini-projet récapitulatif.

Pourquoi c'est important

Un projet Unreal C++ qui ignore les conventions et abuse du Tick devient lent à compiler, lent à exécuter et difficile à maintenir. Cette dernière leçon rassemble les réflexes professionnels, puis tu les appliques dans un mini-jeu qui mobilise toute la piste.

Théorie en profondeur

1. Conventions de nommage Unreal

PréfixePourExemple
AClasse dérivée d'AActorAHeros
UClasse dérivée d'UObject (components…)USante
FStructure / classe simple (hors UObject)FStatsArme
EÉnumérationEEtatIA
bBooléenbEstVivant

Méthodes et propriétés en PascalCase. Ces conventions ne sont pas optionnelles : UHT et l'éditeur s'appuient dessus.

2. Maîtriser le Tick

Le Tick s'exécute chaque frame sur chaque Actor qui l'active — multiplié par des milliers d'objets, c'est un gouffre à performances. Règles :

  • Désactive le Tick si tu n'en as pas besoin : PrimaryActorTick.bCanEverTick = false;.
  • Préfère les événements (CPP13) et les timers (CPP14) à un Tick qui surveille en boucle.
  • Si un Tick est nécessaire mais pas à 60 Hz, augmente son intervalle : PrimaryActorTick.TickInterval = 0.2f;.
// ❌ Tick qui sonde en permanence
void Tick(float Dt) { if (Joueur && Distance(Joueur) < 200) Attaquer(); }

// ✅ Réagir à un événement / une collision quand ça se produit (pas chaque frame)
void OnPlayerOverlap(...) { Attaquer(); }

3. IsValid partout

Le réflexe de sécurité mémoire (CPP10) : vérifie tout pointeur d'objet avant de l'utiliser.

if (IsValid(Cible))
{
    Cible->TakeDamage(/*...*/);
}

4. Compilation : forward declarations & includes

Inclure un .h lourd partout ralentit énormément la compilation. Dans un en-tête, si tu n'utilises qu'un pointeur vers une classe, déclare-la en avant plutôt que de l'inclure ; mets le #include dans le .cpp :

// Heros.h
class UWeaponComponent;          // forward declaration (pas d'include lourd)

UPROPERTY() TObjectPtr<UWeaponComponent> Arme;   // un pointeur suffit

// Heros.cpp
#include "WeaponComponent.h"     // l'include réel va ici

5. Déboguer proprement avec UE_LOG

DECLARE_LOG_CATEGORY_EXTERN(LogJeu, Log, All);   // catégorie dédiée (dans un .h)
DEFINE_LOG_CATEGORY(LogJeu);                      // dans un .cpp

UE_LOG(LogJeu, Warning, TEXT("PV du héros : %d"), PV);
// Affichage écran ponctuel (debug) :
GEngine->AddOnScreenDebugMessage(-1, 2.f, FColor::Green, TEXT("Touché !"));

Crée des catégories de log par système pour filtrer ; et nettoie les messages de debug avant une livraison.

🏁 Mini-projet récapitulatif

Réalise un petit « arène survival » mobilisant toute la piste :

  1. Héros ACharacter déplaçable via Enhanced Input (CPP12), avec un composant USante séparé (CPP7, CPP9).
  2. Ennemis instanciés par un ASpawner sur timer toutes les 2 s (CPP14), stockés dans un TArray (CPP11).
  3. Une interface IDamageable implémentée par le héros et les ennemis (CPP13).
  4. Un delegate OnVieChangee (BlueprintAssignable) mettant à jour une UI Blueprint sans couplage.
  5. Toute référence d'UObject en UPROPERTY(), vérifs IsValid, Tick désactivé là où inutile.
  6. Expose au Blueprint un OnMort() (BlueprintImplementableEvent) pour le feedback visuel.

⚠️ Pièges & erreurs courantes

SymptômeFPS qui s'effondre avec beaucoup d'Actors.
CauseTick actif et coûteux sur des milliers d'objets.

SymptômeCompilation interminable.
CauseIncludes lourds dans les en-têtes.

SymptômeCrashes mémoire intermittents.

SymptômeUne classe « fait tout », impossible à maintenir.

🔧Exercice pratique

  1. Réalise le mini-projet « arène survival » ci-dessus, ou au minimum les points 1 à 4.
  2. Passe ton code en revue : préfixes corrects ? Tick justifié ? IsValid partout ?
  3. Déplace un include lourd d'un .h vers le .cpp via une forward declaration.
Critères de réussite
  • Aucun Tick inutile ; la logique réactive passe par événements/timers.
  • Toutes les références d'UObject sont des UPROPERTY() et vérifiées.
  • Conventions Unreal respectées ; projet organisé et compilant proprement.

Mets en place une catégorie de log dédiée et logge une valeur avec.

// .h
DECLARE_LOG_CATEGORY_EXTERN(LogJeu, Log, All);

UE_LOG(LogJeu, Warning, TEXT("PV du héros : %d"), PV);
📌À retenir
  • Respecte les préfixes (A/U/F/E/I/b) et le PascalCase.
  • Le Tick est coûteux : désactive-le, espace-le, ou remplace-le par événements/timers.
  • Mémoire : UPROPERTY() + IsValid ; compilation : forward declarations.
  • Découpe en composants à rôle unique reliés par delegates ; logue par catégories.

Glossaire des termes introduits

Préfixes Unreal — A/U/F/E/I/b imposés par convention et par UHT.

Forward declaration — annoncer une classe sans l'inclure, pour alléger un en-tête.

TickInterval — espacement (en secondes) entre deux Tick d'un Actor.

Catégorie de log — canal nommé pour filtrer les messages UE_LOG.
