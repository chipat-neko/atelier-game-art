GD07 — Le cycle de vie des nodes

Intermédiaire 45 min Piste 9 · GDScript (Godot)
PrérequisGD6 — Fonctions et GD1.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Identifier les fonctions de cycle de vie : _ready, _process, _physics_process, _input.
  • Utiliser delta pour un mouvement indépendant du framerate.
  • Distinguer rendu (_process) et physique (_physics_process).
  • Réagir aux entrées dans _input.

Pourquoi c'est important

Le moteur appelle automatiquement certaines fonctions « magiques » de tes nodes, à des moments précis. Savoir quoi mettre où (initialisation, logique par frame, physique) est la base d'un gameplay correct et performant — l'équivalent du cycle MonoBehaviour d'Unity (CS7).

Théorie en profondeur

1. Les fonctions de cycle de vie

FonctionQuandUsage typique
_ready()Quand le node (et ses enfants) entre dans la scène.Initialisation, récupérer des références.
_process(delta)Chaque frame de rendu.Animations, logique non physique.
_physics_process(delta)À pas fixe (60 Hz par défaut).Mouvement physique, collisions.
_input(event)À chaque événement d'entrée.Réagir à une touche/clic ponctuel.

2. _ready : l'initialisation

func _ready():
    print("Node prêt")
    # ici, les nodes enfants existent : on peut les référencer

3. delta : indépendance au framerate

delta est le temps (en secondes) écoulé depuis la frame précédente. Multiplier un déplacement par delta garantit la même vitesse quel que soit le nombre d'images par seconde :

var vitesse = 200.0

func _process(delta):
    position.x += vitesse * delta    # avance de 200 px PAR SECONDE, pas par frame
💡Toujours × delta

Sans * delta, la vitesse dépendrait du framerate : rapide sur un PC puissant, lent sur un faible. La règle : tout ce qui évolue dans le temps se multiplie par delta.

4. _process vs _physics_process

_process tourne à chaque frame de rendu (variable). _physics_process tourne à pas fixe (stable) : c'est là que va tout ce qui touche à la physique et au mouvement des corps (move_and_slide, vu en GD12).

5. _input : réagir à un événement

func _input(event):
    if event.is_action_pressed("saut"):   # action définie dans l'Input Map (GD12)
        print("Saut !")

🖐Essaie maintenant~1 min

Mets print("ready") dans _ready() et print(delta) dans _process(delta), puis lance. « ready » n'apparaît qu'une fois ; delta défile à chaque frame : tu vois init vs boucle.

🔧 Démonstration pas à pas

  1. Logge un message dans _ready().
  2. Dans _process(delta), fais glisser un node 2D : position.x += 200 * delta.
  3. Ajoute un _physics_process(delta) et observe la différence de cadence.
  4. Dans _input(event), affiche un message quand une touche est pressée.

⚠️ Pièges & erreurs courantes

SymptômeVitesse différente selon le PC.

CorrectionMultiplie par delta.
SymptômeMouvement physique saccadé.
CausePhysique gérée dans _process.
CorrectionDéplace-la dans _physics_process.
SymptômeRéférence à un enfant null dans _ready.

SymptômeMon node ne bouge pas.

CorrectionRespecte _process(delta) / _physics_process(delta).

🔧Exercice pratique

  1. Un node 2D qui se déplace à vitesse réglable, framerate-indépendant.
  2. Une rotation continue dans _process (× delta).
  3. Un message au pressage d'une touche via _input.
Critères de réussite
  • Le mouvement est cohérent quel que soit le framerate.
  • La physique (si présente) est dans _physics_process.
  • L'initialisation est dans _ready.

Glossaire des termes introduits

_ready — appelée quand le node entre dans l'arbre de la scène.

_process / _physics_process — boucle de rendu (variable) / boucle physique (pas fixe).

delta — temps écoulé depuis la frame précédente (secondes).

_input — fonction appelée à chaque événement d'entrée.
