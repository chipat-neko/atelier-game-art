GD13 — Ressources & variables exportées

Avancé 40 min Piste 9 · GDScript (Godot)
PrérequisGD2 et GD8.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Exposer des variables dans l'Inspector avec @export (et ses variantes).
  • Utiliser @onready pour les références de nodes.
  • Créer une Resource personnalisée pour regrouper des données.
  • Partager et réutiliser des données via des fichiers .tres.

Pourquoi c'est important

Séparer les données du code rend un jeu bien plus maniable : un designer règle les stats d'une arme dans l'Inspector sans toucher au script. Les Resource de Godot sont l'outil idéal pour ça — l'équivalent des ScriptableObject d'Unity (CS15).

Théorie en profondeur

1. @export et ses variantes

@export var vitesse: float = 300.0
@export var nom: String = "Slime"
@export var couleur: Color = Color.RED

@export_range(0, 100) var vie: int = 100          # curseur borné
@export_enum("Facile", "Normal", "Difficile") var difficulte: int
@export_file("*.png") var texture_path: String     # sélecteur de fichier
@export var cible: NodePath                         # référence à un autre node

On peut grouper visuellement avec @export_group("Combat") pour organiser l'Inspector.

2. @onready : les références de nodes

Rappel (GD9) : @onready assigne une variable juste avant _ready, parfait pour récupérer un node enfant. À ne pas confondre avec @export (réglage dans l'Inspector) :

@onready var sprite: Sprite2D = $Sprite2D       # référence interne (chemin)
@export var degats: int = 10                    # réglage exposé au designer

3. Les Resource personnalisées

Une Resource est un conteneur de données sauvegardable (.tres), indépendant de la scène. On en crée une en étendant Resource :

class_name StatsArme
extends Resource

@export var nom: String = "Pistolet"
@export var degats: int = 10
@export var cadence: float = 0.2
@export var portee: float = 800.0

Dans l'éditeur : clic droit ▸ Create New Resource ▸ StatsArme, règle les champs, sauvegarde en pistolet.tres. Puis un node l'utilise :

extends Node2D

@export var stats: StatsArme            # glisse pistolet.tres dans l'Inspector

func _ready():
    print(stats.nom, " fait ", stats.degats, " dégâts")

4. Pourquoi c'est puissant

Plusieurs armes partagent la même structure : tu crées pistolet.tres, fusil.tres, sniper.tres — autant de jeux de données réutilisables, modifiables sans code, et échangeables à chaud. Tu peux aussi charger une Resource par code avec load("res://pistolet.tres").

🔧 Démonstration pas à pas

  1. Expose plusieurs variables avec @export et une variante (@export_range).
  2. Crée une Resource StatsArme (extends Resource) avec des champs @export.
  3. Crée deux .tres (pistolet, fusil) avec des valeurs différentes.
  4. Un node @export var stats: StatsArme : glisse un .tres et affiche ses données.

⚠️ Pièges & erreurs courantes

SymptômeMa Resource n'apparaît pas dans « Create New Resource ».
Causeclass_name manquant sur la Resource.
CorrectionAjoute class_name StatsArme et sauvegarde le script.
SymptômeModifier une instance change toutes les armes.
CauseLa même Resource est partagée par référence.
CorrectionDuplique la Resource (.duplicate()) si tu veux une copie indépendante.
Symptôme@export n'apparaît pas dans l'Inspector.
CauseAnnotation mal placée ou type non exportable.
CorrectionPlace @export juste avant var ; utilise un type exportable.
SymptômeConfusion @export / @onready.
CauseUsages mélangés.
Correction@export = réglage designer ; @onready = référence interne de node.

🔧Exercice pratique

  1. Une Resource StatsEnnemi (pv, vitesse, agressivite) avec champs exportés.
  2. Deux .tres (slime faible, golem fort).
  3. Un node ennemi qui lit ses stats depuis un @export var stats: StatsEnnemi.
Critères de réussite
  • La Resource apparaît et se crée dans l'éditeur.
  • Les deux .tres ont des valeurs distinctes.
  • Le node lit ses données sans valeur en dur.

📌À retenir
  • @export expose une variable dans l'Inspector (+ variantes : range, enum, file, group).
  • @export = réglage designer ; @onready = référence interne de node.
  • Une Resource (class_name + extends Resource) regroupe des données réutilisables (.tres).
  • Sépare données et code : plusieurs .tres partagent une structure.

Glossaire des termes introduits

@export — annotation exposant une variable dans l'Inspector.

@onready — assigne une variable juste avant _ready (références de nodes).

Resource — conteneur de données sauvegardable (.tres), indépendant des scènes.

Custom resource — Resource personnalisée définie par extends Resource + class_name.
