BP10 — Communication entre Blueprints

Avancé 45 min Piste 12 · Blueprints
PrérequisBP9 — Classes & héritage.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Appeler une fonction d'un autre Blueprint via une référence directe.
  • Utiliser Cast To et comprendre son coût.
  • Découpler avec une Blueprint Interface.
  • Diffuser un événement avec un Event Dispatcher.

Pourquoi c'est important

Un jeu, c'est plein d'objets qui doivent se parler : un bouton ouvre une porte, un ennemi prévient le manager qu'il est mort. Choisir le bon canal de communication évite le code spaghetti.

Théorie en profondeur

1. Référence directe

Si un Blueprint possède une référence vers un autre, il peut appeler ses fonctions et lire/écrire ses variables (publiques).

(Porte est une variable BP_Porte)
Bouton ─exec▶ Porte → Ouvrir()    -- appel direct via la référence

2. Cast To

Quand on a une référence générique (ex. Actor), Cast To BP_Porte vérifie si l'objet est bien de ce type et donne accès à son interface. Si le cast échoue, la branche Cast Failed se déclenche. Les casts vers des Blueprints créent une dépendance (chargement) : à utiliser avec mesure.

OtherActor ─▶ Cast To BP_Porte
   ├ (réussi)  ─▶ As BP_Porte → Ouvrir()
   └ Cast Failed ─▶ (ignorer)

3. Blueprint Interface (découplage)

Une Interface déclare des fonctions sans implémentation. Tout Blueprint qui l'implémente peut recevoir le message — sans avoir besoin de connaître son type exact. C'est le moyen propre de communiquer sans Cast.

Interface "BPI_Interactable" : fonction Interagir()
BP_Porte, BP_Coffre, BP_PNJ  implémentent Interagir() chacun à leur façon

Joueur ─▶ (Cible) → Interagir() via interface   -- pas besoin de caster

4. Event Dispatcher (diffusion)

Un Event Dispatcher permet à un Blueprint d'annoncer un événement ; d'autres s'y abonnent (Bind) et réagissent. Communication un-vers-plusieurs, du bas vers le haut (ex. un item annonce « ramassé » au système d'inventaire).

🔧 Démonstration pas à pas

  1. Donne au bouton une référence vers la porte et appelle Ouvrir().
  2. Depuis un Actor générique, fais un Cast To et gère l'échec.
  3. Crée une interface Interagir() et implémente-la dans deux objets.
  4. Crée un Event Dispatcher « Ramassé » et abonne un manager.

⚠️ Pièges & erreurs courantes

Symptôme« Accessed None » au runtime.
CauseRéférence non assignée (vide).
CorrectionAssigne la référence avant de l'utiliser ; teste sa validité (IsValid).
SymptômeTemps de chargement qui gonfle.
CauseCasts en cascade créant des dépendances dures.
CorrectionPréfère les interfaces pour découpler.
SymptômeTout dépend de tout.
CauseRéférences directes partout.

SymptômeL'abonné ne réagit pas.
CauseOubli du Bind au Dispatcher.
CorrectionBind l'événement (souvent au BeginPlay).

🔧Exercice pratique

  1. Bouton → porte via référence directe.
  2. Interface Interagir() implémentée par porte et coffre.
  3. Item qui diffuse « Ramassé » via Dispatcher à un manager.
Critères de réussite
  • Tu gères l'échec d'un Cast.
  • Tu utilises une interface pour découpler.
  • Ton abonné est bien Bind au Dispatcher.

📌À retenir
  • Référence directe : simple mais couplé.
  • Cast To : vérifie le type (gère Cast Failed) ; crée une dépendance.
  • Interface : communiquer sans connaître le type — découplé.
  • Event Dispatcher : diffusion un-vers-plusieurs (Bind pour s'abonner).

Glossaire des termes introduits

Cast To — vérifie/convertit une référence vers une classe précise.

Blueprint Interface — contrat de fonctions implémentées par plusieurs Blueprints.

Event Dispatcher — événement diffusé auquel d'autres Blueprints s'abonnent (Bind).

Accessed None — erreur d'accès à une référence vide.
