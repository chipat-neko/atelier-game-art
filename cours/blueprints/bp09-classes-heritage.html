BP09 — Classes & héritage de Blueprint

Intermédiaire 40 min Piste 12 · Blueprints
PrérequisBP8 — Fonctions & macros.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Le rôle de la classe parente et créer un Blueprint enfant.
  • Surcharger (override) un événement ou une fonction héritée.
  • Utiliser Parent: pour appeler la version parente.
  • Comprendre le Construction Script.

Pourquoi c'est important

L'héritage évite de tout réécrire : un Blueprint BP_Ennemi définit le comportement commun, et BP_Gobelin, BP_Orc en héritent en ne changeant que ce qui diffère. Mêmes principes que la POO du C++ ou de C#.

Théorie en profondeur

1. Classe parente & enfant

Tout Blueprint dérive d'une classe parente (souvent Actor, Pawn, Character). On crée un enfant via clic droit sur le Blueprint → Create Child Blueprint Class. L'enfant hérite variables, composants et logique du parent.

2. Surcharge (override)

Dans l'enfant, on peut surcharger un événement/fonction du parent pour le spécialiser. Les fonctions héritées apparaissent dans My Blueprint (survol → Override).

BP_Ennemi          : fonction Attaquer() → dégâts de base
   └ BP_Gobelin     : override Attaquer() → dégâts + recule
   └ BP_Orc         : override Attaquer() → gros dégâts, lent

3. Appeler le parent (Parent:)

Dans une fonction surchargée, le nœud Parent: NomFonction exécute la logique du parent — pratique pour compléter sans tout réécrire.

Override Attaquer ─exec▶ Parent: Attaquer ─exec▶ (ajouter effet spécifique)

4. Construction Script

Le Construction Script s'exécute à la construction de l'Actor (dans l'éditeur et au spawn), avant le jeu. Idéal pour configurer l'apparence selon des variables exposées (ex. choisir un mesh selon un type). Différent de BeginPlay (qui, lui, démarre le gameplay).

🖐Essaie maintenant~1 min

Fais un clic droit sur un Blueprint d'acteur ▸ Create Child Blueprint Class. Qu'hérite l'enfant ? (variables, composants et logique du parent, que tu peux surcharger sans toucher au parent.)

🔧 Démonstration pas à pas

  1. Crée BP_Ennemi avec une fonction Attaquer().
  2. Crée BP_Gobelin en enfant et surcharge Attaquer().
  3. Appelle Parent: Attaquer puis ajoute un effet.
  4. Configure l'apparence dans le Construction Script.

⚠️ Pièges & erreurs courantes

SymptômeModifier l'enfant casse aussi le parent.
CauseLogique commune mise dans l'enfant.
CorrectionPlace le commun dans le parent, le spécifique dans l'enfant.
SymptômeL'override ignore la logique parente.
CauseOubli du nœud Parent:.
CorrectionAppelle Parent: NomFonction si tu veux la conserver.
SymptômeDu gameplay tourne dans l'éditeur.
CauseLogique de jeu mise dans le Construction Script.
CorrectionLe gameplay va dans BeginPlay ; la config visuelle dans Construction Script.
SymptômeMauvaise classe parente choisie.
CausePawn au lieu de Character, etc.
CorrectionChoisis le parent adapté (Reparent possible mais délicat).

🔧Exercice pratique

  1. Crée un parent BP_Item (variable Nom, fonction Ramasser()).
  2. Crée BP_Potion en enfant et surcharge Ramasser().
  3. Dans le Construction Script, change la couleur selon une variable.
Critères de réussite
  • Le commun est dans le parent, le spécifique dans l'enfant.
  • Tu utilises Parent: quand c'est pertinent.
  • Tu distingues Construction Script et BeginPlay.

Exercices

Exercice 1Prise en main

Quelle classe parente pour un personnage contrôlable ?

Voir une solution

Character (un Pawn spécialisé avec mouvement et capsule). Pour un objet simple non contrôlé, Actor suffit.

Exercice 2Application

Décris l'override d'Attaquer() qui complète le parent.

Voir une piste 
Override Attaquer
   ─exec▶ Parent: Attaquer        (dégâts de base)
   ─exec▶ Play Sound "cri"        (effet propre à l'enfant)
Exercice 3Défi

Conçois une hiérarchie d'items (parent + 2 enfants) et liste ce qui est partagé.

Voir une piste

Parent BP_Item : variables (Nom, Icone), composant Mesh, fonction Ramasser() commune. Enfants BP_Potion (soigne) et BP_Cle (déverrouille) surchargent Ramasser(). Le mesh, l'UI de ramassage et la logique de base restent dans le parent.

📌À retenir
  • Un enfant hérite variables, composants et logique du parent.
  • Override pour spécialiser ; Parent: pour réutiliser la version parente.
  • Commun → parent ; spécifique → enfant.
  • Construction Script = apparence/config ; BeginPlay = gameplay.

Glossaire des termes introduits

Classe parente — classe dont un Blueprint dérive (Actor, Pawn, Character…).

Blueprint enfant — Blueprint héritant d'un parent.

Override — redéfinir un événement/fonction hérité.

Construction Script — graphe exécuté à la construction de l'Actor (apparence).
