CS9 — Héritage, interfaces & polymorphisme

Avancé 45 min Piste 7 · C# (Unity)
PrérequisCS8 — Classes & POO.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Faire hériter une classe d'une autre et réutiliser son code.
  • Redéfinir un comportement avec virtual/override (et abstract).
  • Définir et implémenter une interface (un contrat).
  • Exploiter le polymorphisme pour traiter des objets différents de façon uniforme.

Pourquoi c'est important

Tes ennemis partagent beaucoup : vie, déplacement, mort. Plutôt que de tout copier dans chaque script, l'héritage permet de mettre le commun dans une classe de base (Ennemi) et de ne spécialiser que ce qui change (Gobelin, Dragon). Les interfaces et le polymorphisme, eux, te laissent écrire du code générique (« tout ce qui peut subir des dégâts ») sans connaître le type exact. C'est ce qui rend un gros projet maintenable.

Théorie en profondeur

1. Héritage : réutiliser une classe de base

// Classe de base (parent)
public class Ennemi : MonoBehaviour
{
    public int vie = 100;
    protected float vitesse = 3f;   // 'protected' : visible des classes filles

    public void SubirDegats(int montant)
    {
        vie -= montant;
        if (vie <= 0) Mourir();
    }

    protected virtual void Mourir()   // 'virtual' : redéfinissable par les filles
    {
        Debug.Log("L'ennemi meurt.");
        Destroy(gameObject);
    }
}

// Classe fille : HÉRITE de tout Ennemi, et ajoute/modifie
public class Gobelin : Ennemi
{
    protected override void Mourir()  // on REDÉFINIT le comportement
    {
        Debug.Log("Le gobelin lâche une pièce d'or !");
        base.Mourir();                // appelle la version du parent
    }
}

Points clés : Gobelin : Ennemi = « Gobelin est un Ennemi ». protected = accessible par la classe et ses filles (entre private et public). virtual autorise la redéfinition ; override la réalise ; base.Mourir() appelle la version parente.

À l'inverse de virtual, le mot-clé sealed interdit d'aller plus loin : une classe sealed class Boss : Ennemi ne peut plus servir de parent, et un sealed override bloque toute redéfinition ultérieure. On « scelle » une classe quand on veut garantir que son comportement ne sera pas modifié par héritage (sécurité et, accessoirement, petite optimisation).

2. abstract : forcer une implémentation

Une classe/méthode abstract définit un contrat sans corps que les filles doivent remplir. On ne peut pas instancier une classe abstraite :

public abstract class Arme : MonoBehaviour
{
    public abstract void Attaquer();   // pas de corps : chaque arme DOIT le définir
}

public class Epee : Arme
{
    public override void Attaquer() { Debug.Log("Coup d'épée"); }
}

3. Interfaces : un contrat pur

Une interface (par convention préfixée I) liste des méthodes/propriétés que la classe s'engage à fournir, sans code. Une classe peut implémenter plusieurs interfaces (alors qu'elle n'hérite que d'une classe) :

public interface IDamageable
{
    void SubirDegats(int montant);   // contrat : « je sais subir des dégâts »
}

// Le joueur ET les ennemis peuvent l'implémenter, même sans lien de parenté :
public class Joueur : MonoBehaviour, IDamageable
{
    public void SubirDegats(int montant) { /* ... */ }
}
public class Tonneau : MonoBehaviour, IDamageable
{
    public void SubirDegats(int montant) { /* se casse */ }
}

4. Polymorphisme : traiter le différent de façon uniforme

Le polymorphisme (« plusieurs formes ») permet de manipuler des objets variés via leur type de base ou une interface commune. Une balle n'a pas besoin de savoir si elle touche un joueur, un ennemi ou un tonneau : si c'est un IDamageable, elle l'abîme :

// 'cible' peut être N'IMPORTE QUEL objet qui implémente IDamageable
void Toucher(IDamageable cible)
{
    cible.SubirDegats(20);   // chaque type réagit À SA façon
}

C'est l'intérêt majeur : ajouter un nouvel ennemi qui implémente IDamageable fonctionne sans modifier le code de la balle. (En Unity, on récupère souvent l'interface avec GetComponent<IDamageable>() — CS11.)

🖐Essaie maintenant~1 min

Fais class Ennemi { public virtual void Cri() => Debug.Log("..."); } et class Boss : Ennemi { public override void Cri() => Debug.Log("GROAR"); }. Appelle Cri() sur un Boss : quelle version s'exécute ? (celle du Boss.)

🔧 Démonstration pas à pas

  1. Crée une classe Ennemi : MonoBehaviour avec vie, SubirDegats, et un virtual void Mourir().
  2. Crée Gobelin : Ennemi qui override Mourir() pour afficher un message spécifique puis base.Mourir().
  3. Définis interface IDamageable { void SubirDegats(int m); }, fais Ennemi l'implémenter.
  4. Écris une méthode void Frapper(IDamageable cible) et appelle-la sur un Gobelin : le polymorphisme fait le reste.

⚠️ Pièges & erreurs courantes

SymptômeMa redéfinition n'est pas prise en compte.
CauseMéthode parent non virtual, ou fille sans override (tu as « caché » la méthode).
CorrectionMarque le parent virtual/abstract et la fille override.
Symptôme« does not implement interface member ».
CauseTu déclares implémenter une interface mais une méthode du contrat manque (ou signature différente).
CorrectionImplémente exactement chaque membre de l'interface (nom, type, paramètres).
SymptômeJe veux hériter de deux classes.
CauseC# n'autorise qu'une seule classe de base (héritage simple).
CorrectionUtilise des interfaces (plusieurs possibles) pour combiner des capacités.
Symptôme« Cannot create an instance of the abstract class ».
CauseTu instancies une classe abstract.
CorrectionInstancie une classe fille concrète qui la complète.

🔧Exercice pratique

  1. Classe de base Vehicule avec vitesse et virtual void Demarrer() ; deux filles Voiture et Moto qui override.
  2. Interface IRamassable { void Ramasser(); } implémentée par une Piece et un Coffre.
  3. Méthode void RamasserTout(IRamassable[] objets) qui appelle Ramasser() sur chacun (polymorphisme + foreach).
Critères de réussite
  • Les filles réutilisent et spécialisent la base (override + base).
  • Des types sans lien de parenté partagent une interface.
  • Ton code générique traite des objets variés via l'interface.

📌À retenir
  • Héritage (:) = « est un » ; protected, virtual/override, base.X(), abstract (contrat obligatoire).
  • Interface (I…) = « sait faire », un contrat pur ; une classe en implémente plusieurs (héritage simple, interfaces multiples).
  • Polymorphisme : manipuler des objets variés via leur base/interface → code extensible sans modification.

Glossaire des termes introduits

Héritage — une classe fille réutilise et spécialise une classe de base.

virtual / override — autoriser puis réaliser la redéfinition d'une méthode.

abstract — classe/méthode sans corps, à compléter obligatoirement par les filles.

Interface — contrat de méthodes qu'une classe s'engage à fournir (préfixe I).

Polymorphisme — traiter des objets de types différents via un type commun.
