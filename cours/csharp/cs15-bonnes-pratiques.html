CS15 — Bonnes pratiques, performances & organisation

Avancé 45 min Piste 7 · C# (Unity)
Prérequisidéalement toute la piste, surtout CS11, CS13 et CS14.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Appliquer les conventions C#/Unity (nommage, SerializeField).
  • Respecter la responsabilité unique (un script = un rôle).
  • Éviter les pièges de performance (Update, allocations, GetComponent/Find).
  • Organiser un projet et réaliser un mini-projet récapitulatif.

Pourquoi c'est important

Un jeu qui « marche » mais dont le code est un plat de spaghettis devient vite impossible à faire évoluer : chaque ajout casse autre chose, les bugs se multiplient, les perfs s'effondrent. Cette dernière leçon rassemble les réflexes qui font la différence entre un script jetable et un projet maintenable — puis tu mets tout en pratique dans un mini-jeu.

Théorie en profondeur

1. Conventions de nommage

public class JoueurController : MonoBehaviour   // PascalCase pour classes/méthodes
{
    public float VitesseMax = 5f;               // PascalCase pour public
    [SerializeField] private int pointsDeVie;   // camelCase pour private
    private const int MAX_VIE = 100;            // constantes en MAJUSCULES

    private void AppliquerDegats(int montant) { }  // PascalCase, verbe d'action
}

Des noms clairs et cohérents valent mieux que des commentaires : vitesseDeplacement plutôt que v.

2. SerializeField plutôt que public

Pour régler une valeur dans l'Inspector sans casser l'encapsulation (CS8), expose le champ avec [SerializeField] private au lieu de public :

[SerializeField] private float vitesse = 5f;   // visible dans l'Inspector, protégé du code externe
[Header("Combat")]                              // titre de section dans l'Inspector
[SerializeField, Range(0, 100)] private int degats = 10;  // curseur borné

3. Responsabilité unique

Un script qui gère à la fois le déplacement, la vie, l'UI, le son et la sauvegarde est ingérable. Découpe : Deplacement, Sante, InventaireUI… Chacun fait une chose, et ils communiquent par événements (CS13) ou références sérialisées.

4. Performances : les réflexes essentiels

  • Cache les GetComponent dans Awake/Start — jamais dans Update (CS11).
  • Évite GameObject.Find et Camera.main répétés en boucle : stocke la référence.
  • Pas d'allocations par frame : pas de new inutile, pas de string concaténées dans Update (elles font travailler le ramasse-miettes → micro-freezes).
  • Object pooling : au lieu d'Instantiate/Destroy en boucle (balles, ennemis), réutilise un pool d'objets désactivés/réactivés.
  • Désactive ce qui n'a rien à faire : un Update vide a quand même un coût × milliers d'objets.
// ❌ alloue une chaîne à chaque frame → pression sur le GC
void Update() { label.text = "Score : " + score; }

// ✅ ne met à jour QUE quand ça change (via un événement, CS13)
void OnScoreChange(int s) { label.text = "Score : " + s; }

5. Organisation du projet

Range tes assets : Scripts/, Prefabs/, Materials/, Scenes/, Audio/… Un script par fichier, nommé comme sa classe. Pour les données partagées (stats d'armes, niveaux), regarde les ScriptableObject — des assets de données réutilisables, sans GameObject.

6. Déboguer proprement

Debug.Log pour comprendre le flux, mais nettoie-les avant de livrer (ils coûtent en perf). Utilise les points d'arrêt de l'IDE, et Debug.DrawRay/Gizmos pour visualiser dans la Scene.

🏁 Mini-projet récapitulatif

Réalise un petit « survival » qui mobilise toute la piste :

  1. Joueur déplaçable au clavier via Rigidbody (CS12), avec un script Sante séparé (CS8).
  2. Ennemis instanciés par un spawner en coroutine, toutes les 2 s (CS14, CS11), stockés dans une List (CS10).
  3. Une interface IDamageable implémentée par le joueur et les ennemis (CS9).
  4. Un événement OnScoreChange qui met à jour l'UI sans couplage (CS13).
  5. Un menu pause via Time.timeScale et un Game Over via OnMort.
  6. Applique les bonnes pratiques : SerializeField, références en cache, scripts à responsabilité unique, dossiers rangés.

🔧Exercice pratique

  1. Réalise le mini-projet « survival » ci-dessus, ou au minimum les points 1 à 4.
  2. Passe ton code en revue : chaque script a-t-il un seul rôle ? Y a-t-il un GetComponent/Find dans un Update ?
  3. Remplace une boucle Instantiate/Destroy par un petit pool d'objets réutilisés.
Critères de réussite
  • Aucun GetComponent/Find dans un Update.
  • Les scripts ont des rôles distincts et communiquent par événements.
  • Les réglages passent par [SerializeField], le projet est rangé en dossiers.

Exercices

Exercice 1Échauffement

Reprends un script avec des champs public « pour les voir dans l'Inspector » et convertis-les en [SerializeField] private.

Voir une solution 
// Avant
public float vitesse = 5f;

// Après : visible dans l'Inspector, protégé du code externe
[SerializeField] private float vitesse = 5f;
Exercice 2Application

Remplace une mise à jour d'UI faite chaque frame dans Update par une mise à jour déclenchée seulement quand la valeur change (via un événement).

Voir une solution 
// ❌ alloue une string à chaque frame
void Update() { label.text = "Score : " + score; }

// ✅ ne met à jour QUE sur changement
void OnScoreChange(int s) { label.text = "Score : " + s; }
Exercice 3Défi

Esquisse un mini « pool » de balles : au lieu d'Instantiate/Destroy, réactive une balle inactive d'une liste.

Voir une solution 
using System.Collections.Generic;

private List<GameObject> pool = new List<GameObject>();

GameObject ObtenirBalle()
{
    foreach (GameObject b in pool)
        if (!b.activeInHierarchy) { b.SetActive(true); return b; }   // réutilise

    GameObject nouvelle = Instantiate(prefabBalle);   // sinon en crée une
    pool.Add(nouvelle);
    return nouvelle;
}
// Au lieu de Destroy : balle.SetActive(false);
📌À retenir
  • Nomme clairement ; expose avec [SerializeField] private, pas public.
  • Un script = un rôle ; ils communiquent par événements (faible couplage).
  • Perf : cache les références, évite les allocations/Find par frame, pense au pooling.
  • Range le projet, nettoie les Debug.Log, et teste régulièrement.

Glossaire des termes introduits

SerializeField — attribut exposant un champ privé dans l'Inspector.

Responsabilité unique — principe : chaque script/classe n'a qu'un seul rôle.

Object pooling — réutiliser des objets désactivés au lieu d'en créer/détruire sans cesse.

ScriptableObject — asset de données réutilisable, indépendant d'un GameObject.
