CS13 — Événements & communication entre scripts

Avancé 45 min Piste 7 · C# (Unity)
PrérequisCS9 — Héritage & interfaces et CS11.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Comprendre le couplage et pourquoi on cherche à le réduire.
  • Utiliser un delegate / Action pour passer un comportement.
  • Déclarer et déclencher un événement C# (event Action).
  • Connaître l'alternative UnityEvent (branchée dans l'Inspector).

Pourquoi c'est important

Quand le joueur meurt, il faut prévenir l'UI (afficher « Game Over »), le son (jouer un jingle), le score (l'enregistrer), le spawner (arrêter)… Si le script Joueur appelle directement chacun de ces scripts, il devient un nœud qui connaît tout le monde : c'est du couplage fort, difficile à maintenir. Les événements inversent la logique : le Joueur annonce « je suis mort », et qui veut réagir s'abonne. C'est le patron observateur.

Le couplage, concrètement

// ❌ Couplage fort : le Joueur doit connaître TOUS les autres scripts
void Mourir()
{
    ui.AfficherGameOver();
    audio.JouerJingle();
    score.Enregistrer();
    spawner.Arreter();      // ajouter une réaction = modifier ce code à chaque fois
}

Théorie en profondeur

1. Delegate & Action : une variable qui contient une méthode

Un delegate est un type qui référence une méthode. System.Action est un delegate prêt à l'emploi (sans valeur de retour) :

using System;

Action saluer = () => Debug.Log("Bonjour");
saluer();                          // appelle la méthode stockée

Action<int> afficher = n => Debug.Log("PV : " + n);
afficher(100);                     // Action<int> = prend un int

On peut cumuler plusieurs méthodes dans une Action avec += (multicast) : les appeler toutes d'un coup. C'est la base de l'événement.

2. L'événement C# : annoncer, s'abonner

public class Joueur : MonoBehaviour
{
    // L'ÉVÉNEMENT : "quelque chose s'est produit", les autres peuvent s'y abonner
    public event Action OnMort;

    public void Mourir()
    {
        Debug.Log("Le joueur meurt");
        OnMort?.Invoke();   // déclenche → tous les abonnés réagissent (?. = si non-null)
    }
}

Et côté abonnés, chacun s'inscrit/désinscrit :

public class UIGameOver : MonoBehaviour
{
    public Joueur joueur;   // référence (Inspector)

    void OnEnable()  { joueur.OnMort += Afficher; }   // s'abonner
    void OnDisable() { joueur.OnMort -= Afficher; }   // SE DÉSABONNER (important !)

    void Afficher() { Debug.Log("GAME OVER"); }
}

Le mot-clé event protège l'événement : de l'extérieur on peut seulement +=/-=, pas le remplacer ni le déclencher. Toujours se désabonner dans OnDisable/OnDestroy pour éviter les fuites et les appels sur objets détruits.

3. Passer des données avec l'événement

public event Action<int> OnVieChangee;   // transporte la nouvelle valeur

void Subir(int degats)
{
    pv -= degats;
    OnVieChangee?.Invoke(pv);            // l'UI met à jour la barre de vie
}

4. UnityEvent : brancher dans l'Inspector

Alternative « visuelle » : un UnityEvent se câble dans l'Inspector (comme le bouton UI). Pratique pour les designers, sans code côté abonné :

using UnityEngine.Events;

public UnityEvent surMort;   // apparaît dans l'Inspector
// ... puis dans le code : surMort.Invoke();

Quand choisir ? event Action = communication code-à-code, rapide, typée. UnityEvent = câblage dans l'éditeur, flexible pour le level design. SendMessage existe mais est lent et fragile (par nom de méthode) — à éviter.

🔧 Démonstration pas à pas

  1. Dans Joueur, déclare public event Action OnMort; et déclenche-le dans Mourir() avec OnMort?.Invoke();.
  2. Crée UIGameOver qui référence le Joueur, s'abonne dans OnEnable, se désabonne dans OnDisable, et logge « GAME OVER ».
  3. Ajoute un 2ᵉ abonné (un script Audio) sans toucher au Joueur : tu constates le découplage.
  4. Variante typée : event Action<int> OnVieChangee pour mettre à jour une barre de vie.

⚠️ Pièges & erreurs courantes

SymptômeNullReferenceException au déclenchement.
CauseAucun abonné : l'événement vaut null.
CorrectionDéclenche avec l'opérateur conditionnel : OnMort?.Invoke();.
SymptômeErreur « MissingReference » ou réactions en double.
CauseAbonnement jamais retiré (objet détruit ou réactivé).
CorrectionDésabonne-toi (-=) dans OnDisable/OnDestroy, symétrique de l'abonnement.
Symptôme« cannot access event outside the class ».
CauseTu tentes d'Invoke l'event depuis un autre script.
CorrectionSeule la classe propriétaire déclenche ; expose plutôt une méthode publique.
SymptômeCommunication « par nom » qui casse en silence.
CauseUsage de SendMessage("...") (fragile, non typé, lent).
CorrectionPréfère event Action ou UnityEvent.

🔧Exercice pratique

  1. Un système de vie qui déclenche OnVieChangee(int) et OnMort().
  2. Une UI qui s'abonne pour mettre à jour une barre de vie et afficher Game Over.
  3. Un 2ᵉ abonné (audio ou score) ajouté sans modifier le script de vie.
Critères de réussite
  • L'émetteur ne référence aucun abonné (découplage réel).
  • Chaque abonnement (+=) a son désabonnement (-=).
  • Le déclenchement utilise ?.Invoke().

📌À retenir
  • Les événements réduisent le couplage : l'émetteur annonce, les abonnés réagissent (observateur).
  • Action/delegate = une variable contenant une (des) méthode(s) ; event la protège.
  • Déclenche avec OnX?.Invoke() ; abonne avec +=, désabonne avec -=.
  • UnityEvent pour câbler dans l'Inspector ; évite SendMessage.

Glossaire des termes introduits

delegate / Action — type référençant une ou plusieurs méthodes, appelables ensemble.

event — delegate protégé : l'extérieur ne peut que s'abonner/se désabonner.

Observateur (observer) — patron où des abonnés réagissent à un émetteur sans qu'il les connaisse.

UnityEvent — événement sérialisé, câblable dans l'Inspector.
