CS1 — Introduction au C# & premier script

Débutant 35 min Piste 7 · C# (Unity)
PrérequisAucune notion de code requise. Pour pratiquer, il faut Unity installé (renvoi U1) et savoir te repérer dans l'éditeur (renvoi U2).
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Expliquer ce qu'est le C# et ce que fait un script dans Unity.
  • Comprendre qu'un script est un Component via la classe MonoBehaviour.
  • Créer un script, l'ouvrir dans l'éditeur de code, et l'attacher à un GameObject.
  • Écrire ta première ligne (Debug.Log) et comprendre le cycle écrire → compiler → jouer.

Pourquoi c'est important

Les pistes précédentes t'ont appris à fabriquer ce qu'on voit (modèles, textures, lumière). Le code, lui, fabrique ce qui se passe : un personnage qui saute, un ennemi qui poursuit, un score qui monte. Sans code, un jeu n'est qu'un décor figé. En Unity, on programme en C# (prononcé « C sharp »), un langage moderne, lisible et très demandé sur le marché du jeu vidéo. Cette leçon pose la toute première brique : faire exécuter ta ligne de code par le moteur.

Théorie en profondeur

1. C# et .NET, en deux mots

Le C# est un langage de programmation créé par Microsoft. Unity l'utilise comme langage de script : tu écris des instructions en C#, Unity les compile (les traduit en quelque chose que la machine exécute) et les fait tourner pendant le jeu. Tu n'as pas besoin de connaître les rouages de .NET pour commencer : retiens juste que C# est un langage typé (chaque donnée a un type, voir CS2) et orienté objet (tout s'organise en « classes », voir CS8).

2. Un script = un Component (MonoBehaviour)

Rappel de U2 : dans Unity, un GameObject tient ses comportements de ses Components. Un script est un type de Component que tu écris toi-même. Pour qu'Unity le reconnaisse comme tel, ta classe doit hériter de MonoBehaviour (la classe de base de tous les scripts Unity). C'est ce qui te donne accès aux fonctions spéciales appelées automatiquement par le moteur — comme Start() et Update() (détaillées en CS7).

3. Anatomie d'un script par défaut

Quand tu crées un script nommé MonObjet, Unity génère ceci :

using UnityEngine;            // donne accès aux outils d'Unity (Debug, Transform...)

public class MonObjet : MonoBehaviour   // ma classe EST un MonoBehaviour
{
    // Appelée UNE fois, au démarrage de l'objet
    void Start()
    {
        Debug.Log("Bonjour depuis Unity !");
    }

    // Appelée à CHAQUE image (frame)
    void Update()
    {

    }
}

Décortiquons :

  • using UnityEngine; — une directive qui importe la boîte à outils d'Unity. Sans elle, Debug et MonoBehaviour seraient inconnus.
  • public class MonObjet : MonoBehaviour — déclare une classe nommée MonObjet qui hérite de MonoBehaviour (le « : MonoBehaviour » = héritage, voir CS9).
  • void Start() et void Update() — des méthodes (voir CS6). Unity les appelle tout seul aux bons moments. void signifie « ne renvoie rien ».
  • Les accolades { } délimitent un bloc ; chaque instruction se termine par un point-virgule ;.
  • Debug.Log("…") affiche un message dans la Console d'Unity — ton premier outil de diagnostic.
ℹ️Règle d'or du nommage

Le nom du fichier (MonObjet.cs) doit être exactement identique au nom de la classe (MonObjet). Sinon, Unity refuse d'attacher le script. Le C# est aussi sensible à la casse : Start ≠ start.

4. Le cycle écrire → compiler → jouer

À chaque fois que tu modifies et sauvegardes un script, Unity le recompile en arrière-plan (une petite roue tourne en bas à droite). S'il y a une erreur de syntaxe, elle s'affiche en rouge dans la Console et le jeu ne se lancera pas tant qu'elle n'est pas corrigée. Quand tout compile, tu appuies sur Play (▶) et ton code s'exécute.

🔧 Démonstration pas à pas

  1. Dans la fenêtre Project, clic droit dans un dossier Scripts ▸ Create▸C# Script. Nomme-le immédiatement HelloWorld (le renommer plus tard est source d'erreurs).
  2. Double-clique le script : il s'ouvre dans ton éditeur de code (Visual Studio, Rider ou VS Code — installé avec Unity ou via les modules, renvoi U1).
  3. Dans Start(), écris : Debug.Log("Mon premier script !"); puis sauvegarde (Ctrl+S).
  4. Reviens dans Unity : attends la fin de la compilation (la roue en bas à droite). Si la Console affiche une erreur rouge, relis ta ligne (point-virgule, guillemets, casse).
  5. Attache le script : glisse-le depuis le Project sur un GameObject (ou sélectionne l'objet ▸ Add Component ▸ tape « HelloWorld »).
  6. Ouvre la Console (Window ▸ General ▸ Console), appuie sur Play ▸ ton message s'affiche ! Tu viens d'exécuter ton premier code.

⚠️ Pièges & erreurs courantes

Symptôme« Can't add script component… the script class cannot be found. »
CauseLe nom du fichier ne correspond pas au nom de la classe (ex. fichier renommé après création).
CorrectionMets le nom du fichier et le public class … strictement identiques.
SymptômeLe bouton Play est grisé / « All compiler errors must be fixed ».
CauseUne erreur de syntaxe (point-virgule oublié, accolade manquante, faute de casse).
CorrectionDouble-clique l'erreur rouge dans la Console : elle t'emmène à la ligne fautive.
SymptômeRien ne s'affiche dans la Console au Play.
CauseScript non attaché à un objet actif, ou message écrit hors de Start()/Update().

SymptômeMes modifs de code ne changent rien en jeu.
CauseScript non sauvegardé, ou modifié pendant le mode Play (les changements en Play sont annulés à l'arrêt).

🔧Exercice pratique

  1. Crée un script HelloWorld, attache-le à un objet, affiche un message au démarrage avec Debug.Log.
  2. Ajoute un second Debug.Log("…") dans Update() et observe : il s'affiche en boucle (une fois par image). Comprends pourquoi.
  3. Provoque volontairement une erreur (retire un point-virgule), lis le message rouge, puis corrige.
Critères de réussite
  • Ton message s'affiche dans la Console au Play.
  • Tu sais expliquer la différence Start (une fois) vs Update (chaque image).
  • Tu sais lire une erreur de compilation et la corriger.

Exercice 1Échauffement

Crée un script Bonjour, attache-le à un GameObject vide, et affiche « Bonjour Unity ! » au démarrage.

Voir une solution 
using UnityEngine;

public class Bonjour : MonoBehaviour
{
    void Start()
    {
        Debug.Log("Bonjour Unity !");
    }
}
Exercice 2Application

Dans le même script, ajoute un message dans Update() et un compteur d'images affiché toutes les 60 frames (indice : une variable int et l'opérateur %).

Voir une solution 
public class Bonjour : MonoBehaviour
{
    int images = 0;

    void Start()
    {
        Debug.Log("Bonjour Unity !");
    }

    void Update()
    {
        images++;
        if (images % 60 == 0)
            Debug.Log("60 images écoulées : " + images);
    }
}
Exercice 3Défi

Provoque une erreur de compilation (retire un ;), lis le message rouge dans la Console, corrige-la, puis explique en une phrase pourquoi le bouton Play était grisé.

Voir une solution

Le point-virgule manquant casse la syntaxe : la compilation échoue, donc Unity refuse d'entrer en mode Play tant que toutes les erreurs ne sont pas corrigées. Double-cliquer l'erreur rouge dans la Console ouvre la ligne fautive dans l'éditeur de code.

// ❌ erreur : il manque le ;
Debug.Log("Test")

// ✅ corrigé
Debug.Log("Test");

📌À retenir
  • On programme Unity en C# ; un script = un Component que tu écris, via la classe MonoBehaviour.
  • Le nom du fichier = le nom de la classe ; le C# est sensible à la casse ; chaque instruction finit par ;.
  • Start() s'exécute une fois, Update() à chaque image ; Debug.Log affiche dans la Console.
  • Cycle : écrire → sauvegarder → Unity compile → attacher à un objet → Play.

Glossaire des termes introduits

C# — langage de programmation utilisé par Unity pour scripter le comportement du jeu.

Script — fichier de code (.cs) qui devient un Component une fois attaché à un GameObject.

MonoBehaviour — classe de base dont héritent les scripts Unity ; donne accès aux méthodes appelées par le moteur.

Compilation — traduction du code C# en instructions exécutables ; une erreur bloque le Play.

Debug.Log — instruction qui affiche un message dans la Console d'Unity.
