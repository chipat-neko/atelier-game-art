LUA14 — Performance & pièges

Avancé 40 min Piste 11 · Lua
PrérequisLUA7 — Tables, LUA12 — Le scripting de jeu.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Pourquoi local est plus rapide que global.
  • Réduire les allocations en réutilisant les tables.
  • Le rôle du ramasse-miettes (GC) et comment limiter les pics.
  • Le piège de # sur les tables à trous.

Pourquoi c'est important

Un script de jeu tourne 60 fois par seconde. Une allocation inutile par frame = des milliers par minute, donc des saccades quand le GC passe. Quelques réflexes simples suffisent à garder un framerate stable.

Théorie en profondeur

1. local vs global

Une variable globale est une entrée dans une table (_G) : chaque accès est une recherche par clé. Une variable locale vit dans un registre : l'accès est quasi instantané. En boucle chaude, mettre en cache les fonctions globales en locales aide.

local sqrt = math.sqrt      -- cache local d'une fonction très appelée
local function dist(x, y) return sqrt(x*x + y*y) end

2. Réutiliser les tables (limiter les allocations)

-- ❌ alloue une table neuve à CHAQUE frame
function update(dt)
  local v = {x = 0, y = 0}    -- déchet à ramasser
end

-- ✅ réutilise une table existante
local v = {x = 0, y = 0}
function update(dt)
  v.x, v.y = 0, 0             -- on réécrit, on ne réalloue pas
end

3. Le ramasse-miettes (GC)

Lua libère automatiquement la mémoire des valeurs inaccessibles. C'est pratique, mais un gros pic d'allocations provoque une pause GC visible (micro-freeze). On le maîtrise via collectgarbage("step") ou en allouant moins. Mesure d'abord, optimise ensuite.

print(collectgarbage("count") .. " Ko utilisés")   -- surveiller la mémoire
collectgarbage("collect")                          -- forcer un cycle (debug)

4. Le piège du # sur les tables à trous

local t = {1, 2, nil, 4}
print(#t)   -- 4 OU 2 : indéterminé si trou ! (rappel LUA5)

Sur une séquence avec un nil au milieu, # n'est pas fiable. Garde des séquences denses, ou tiens un compteur explicite.

🔧 Démonstration pas à pas

  1. Mets en cache math.sqrt en local et compare.
  2. Transforme une allocation par frame en réutilisation.
  3. Surveille la mémoire avec collectgarbage("count").
  4. Reproduis le piège du # sur une table à trou.

⚠️ Pièges & erreurs courantes

SymptômeMicro-freezes réguliers.
CauseTrop d'allocations par frame → pauses GC.
CorrectionRéutilise les tables ; alloue hors boucle chaude.
SymptômeOptimisation qui ne change rien.
CauseTu optimises sans mesurer.
CorrectionProfile d'abord ; cible le vrai goulot.
Symptôme#t donne une taille fausse.
CauseTrou (nil) dans la séquence.
CorrectionSéquence dense ou compteur explicite.
SymptômeFuite de mémoire qui grossit.

CorrectionLibère les références ; désabonne les événements.

🔧Exercice pratique

  1. Réécris une fonction update pour supprimer son allocation par frame.
  2. Mets en cache une fonction globale fréquente en local.
  3. Affiche la mémoire avant/après un cycle GC.
Critères de réussite
  • Aucune table allouée inutilement dans la boucle chaude.
  • Tu mesures avant d'optimiser.
  • Tu évites # sur des tables à trous.

Exercice 2Application

Supprime l'allocation par frame d'un vecteur de vélocité.

Voir une solution 
local vel = {x = 0, y = 0}        -- alloué une seule fois
function update(dt, dx, dy)
  vel.x, vel.y = dx, dy           -- réécriture, pas de nouvelle table
end
Exercice 3Défi

Implémente un « pool » de projectiles réutilisables.

Voir une piste 
local pool, actifs = {}, {}
local function obtenir()
  local p = table.remove(pool) or {x = 0, y = 0}   -- recycle ou crée
  table.insert(actifs, p)
  return p
end
local function liberer(p)
  table.insert(pool, p)   -- remis dans le pool plutôt que jeté au GC
end

  • Réutilise les tables au lieu d'en allouer une par frame.

Glossaire des termes introduits

ramasse-miettes (GC) — libère automatiquement la mémoire inaccessible.

allocation — création d'une nouvelle valeur (table, chaîne) en mémoire.

pool d'objets — réserve d'objets recyclés pour éviter d'allouer/jeter sans cesse.

_G — table contenant toutes les variables globales.
