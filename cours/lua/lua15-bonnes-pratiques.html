LUA15 — Bonnes pratiques & mini-projet

Avancé 45 min Piste 11 · Lua
Prérequisidéalement les leçons LUA7 à LUA14.
🎯Objectifs d'apprentissage

À la fin de cette leçon, tu sauras :

  • Les conventions Lua (nommage, local partout, modules).
  • Gérer l'état d'un jeu proprement (machine à états).
  • Déboguer avec print et un petit inspect de table.
  • Assembler tout le cours dans un mini-projet récapitulatif.

Pourquoi c'est important

Savoir écrire du Lua ne suffit pas : il faut écrire du Lua maintenable. Cette leçon condense les réflexes pros et les met en œuvre dans un petit projet complet.

Théorie en profondeur

1. Conventions

  • local par défaut : réserve les globales aux rares cas voulus.
  • Nommage : snake_case ou camelCase — choisis-en un et tiens-t'y.
  • Un module par responsabilité qui retourne une table (rappel LUA9).
  • Valide les entrées tôt avec assert (rappel LUA10).

2. Gestion d'état : la machine à états

local Jeu = {etat = "menu"}

local transitions = {
  menu  = function() Jeu.etat = "partie" end,
  partie = function() Jeu.etat = "pause" end,
  pause = function() Jeu.etat = "partie" end,
}

function Jeu.action()
  local t = transitions[Jeu.etat]   -- table-switch : pas de if/elseif géant
  if t then t() end
end

Une table de transitions remplace une cascade de if/elseif et reste lisible quand les états se multiplient.

3. Déboguer : print & inspect

local function inspect(t, indent)
  indent = indent or ""
  for k, v in pairs(t) do
    if type(v) == "table" then
      print(indent .. tostring(k) .. ":")
      inspect(v, indent .. "  ")
    else
      print(indent .. tostring(k) .. " = " .. tostring(v))
    end
  end
end

inspect({nom = "Léa", stats = {pv = 100, mana = 30}})

4. Mini-projet : un combat au tour par tour

-- combat.lua : assemble classes (LUA8), modules (LUA9) et erreurs (LUA10)
local Combattant = {}
Combattant.__index = Combattant

function Combattant.new(nom, pv, attaque)
  return setmetatable({nom = nom, pv = pv, attaque = attaque}, Combattant)
end

function Combattant:frapper(cible)
  assert(cible, "cible manquante")
  cible.pv = math.max(0, cible.pv - self.attaque)
  print(string.format("%s frappe %s (%d PV restants)", self.nom, cible.nom, cible.pv))
end

function Combattant:estVivant() return self.pv > 0 end

local heros  = Combattant.new("Héros", 100, 18)
local gobelin = Combattant.new("Gobelin", 40, 9)

local tour = 1
while heros:estVivant() and gobelin:estVivant() do
  print("-- Tour " .. tour)
  heros:frapper(gobelin)
  if gobelin:estVivant() then gobelin:frapper(heros) end
  tour = tour + 1
end

print((heros:estVivant() and heros.nom or gobelin.nom) .. " remporte le combat !")

Ce projet réunit : tables, classes via métatables, méthodes avec self, string.format, assert, boucle while et le ternaire and/or. C'est l'aboutissement de la piste.

🔧 Démonstration pas à pas

  1. Convertis une cascade de if/elseif d'états en table de transitions.
  2. Écris un inspect récursif pour visualiser une table.
  3. Assemble le mini-projet de combat et fais-le tourner mentalement.

🔧Exercice pratique

  1. Ajoute un état game_over à la machine à états.
  2. Étends le combat avec un sort qui soigne (réutilise :frapper comme modèle).
  3. Range le tout en deux modules : combattant.lua et main.lua.
Critères de réussite
  • Aucune variable globale involontaire.
  • La machine à états reste lisible quand tu ajoutes un état.
  • Ton mini-projet combine classes, modules et gestion d'erreurs.

Glossaire des termes introduits

machine à états — modèle où le jeu est dans un état parmi plusieurs, avec des transitions définies.

table de transitions — table associant chaque état à son action/état suivant.

inspect — fonction de débogage qui affiche récursivement le contenu d'une table.
